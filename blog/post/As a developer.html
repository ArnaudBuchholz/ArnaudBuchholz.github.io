As a developer...

As a developer, you should be proud about your laziness and I even recommend that you maturate it.

If I ever have to make an interview and the recruiter asks me what my main qualities are, I will start by explaining - eye to eye - that I am lazy guy.

Back to my school days, during my university years, I had this fantastic teacher starting the very first hour of its computer science course by explaining how the computer boom is all about - what he called - the "couch run".
I first thought this guy was funny and his unusual theory was a way to make the course less boring.

However, after few years of development, I now fully embrace the concept and I would like to explain why.

Obviously, the following is my opinion and is subject to discussion. Probably my current company would not agree with that but it looks like managers are not complaining with my work...

First of all, a good developer is too lazy to write the same code twice.
Sorry Mister Tesler! Copy and Paste is not a good solution to apply the code everywhere it is needed.
On the contrary, a good developer will angle for reusability and will always write reusable code.
So that the work is not done twice and, more important, it will be fixed only once.

The same way, when using a framework or a library, he will first check if it provides the service that will help him to develop the feature faster.
If somebody spent hours writing (and debugging) this packaged code, it may already contain elements you can rely on.

Also, a good developer is too lazy to neglect the code quality. Indeed, bad code means bugs. Bugs means extra work.
And extra work is the opposite of what a lazy guy is aiming for, right?
Code quality is a wide topic and there are many discussions about it.
In my opinion, it includes:

  • Making sure the code works. It does what it is supposed to do the right way.

  • Making sure the code is understandable. Did you ever face this situation?
    • Manager: You should document your code!
    • You: Nah, it's OK I just finished it and it works. I don't have enough time in my ticket to add documentation that nobody will read anyway...

      • (six months later)
    • Manager: OK, there is a bug here / we need an evolution, can you fix / do it please?
    • You: Well, I need first to check the code, I don't remember how it works... Man! Did I wrote that? ... Why did I do it this way?...
    It means documentation but not only. Some claim that a good code writing prevents the need for documentation. I somewhat agree. However, the truth is that when you have to read a bunch of code, a summary doesn't hurt (considering it is up-to-date).

  • Making sure the code is maintainable. This usually requires experience but, from a general point of view, if you take any assumption when writing the code: make sure you will remember.

That's why a lazy developer also include tests in its development. And because testing is boring and takes time, those tests are mostly automated.
One valuable freebie of automation is non-regression testing.
