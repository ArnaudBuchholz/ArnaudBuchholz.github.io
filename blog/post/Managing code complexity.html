Managing code complexity

Do you brush your teeth? May I ask you why? I mean... if you have a cavity, you go to the dentist and he treats it, right? But still... fortunately... most people brush their teeth. Now let's talk about code: do you care about quality? I mean... if you have a bug, you fix it, right? But what if I tell you that you can manage the code complexity to reduce the number of bugs? As if you were brushing the code !
## Overview When I started the [GPF-JS](https://github.com/ArnaudBuchholz/gpf-js) project, I had quite a clear vision of what I wanted to achieve but I was unsure about the how. Actually, I wanted to improve my knowledge in JavaScript. several goals in mind and quite a clear vision of what . First, I wanted to improve my knowledge in JavaScript. At that time, I had written many scripts and even started to build a sort of library but I wanted to better understand how a big JavaScript project could be designed following the best practices. The first year, I mostly focused in adding features to the library without caring much about the structure and the tests. From the beginning, I wanted the library to address a large audience by making it compatible with several hosts. Naturally, I came up with polyfills concepts. Rather quickly, testing became a necessity. Indeed, as I was looking for the perfect way of doing things, I was constantly refactoring the code... which also means I broke it many times. I also quickly understood that the code became hard to maintain: too many concepts, ... The first big improvement came with TDD. Then came the use of Plato and coverage tools: this drastically changed the way I developped the library. ## TDD ### UI Automation I often get the question regarding TDD and UI automation.
