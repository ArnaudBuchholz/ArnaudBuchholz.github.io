Code analysis

### Linting is necessary... In a previous post, I already talked about the benefits of using [code linting tools](http://gpf-js.blogspot.ca/2013/11/ why-lint-tool-can-reduce-development.html) as it really saves the developer time during the coding phase: * It highlights common mistakes (undeclared variables, unused parameters, use of [hasOwnProperty](https://developer.mozilla.org/en-US/docs/Web/JavaScript/ Reference/Global_Objects/Object/hasOwnProperty), == vs ===...) * It maintains code consistency (formatting, naming conventions...) * It checks code complexity, in particular the [cyclomatic complexity](http:// en.wikipedia.org/wiki/Cyclomatic_complexity) Regarding JavaScript, one of the best options is to apply [JSHint](http://www.jshint.com/). ### ...but not sufficient However, this kind of tool has limits as it tends to apply general validation rules without considering the context in which the code is executed. To illustrate the problem, let me show you the following code: `(function () { "use strict"; /*global $, freezeInterface, updateInterfaceUsingData, unfreezeInterface*/ // Handles the click on a specific button function onClick () { freezeInterface("Please wait, updating information..."); $.ajax("getInformation.aspx", { success: function (data) { updateInterfaceUsingData(data); } }); unfreezeInterface(); } })();` Using the "global" comment, JSHint is configured to be aware of the global variables to be defined. Besides the fact the **onClick** function is not used, the code successfully pass JSHint validation. But do you see any problem here? ### Understanding the code The method **[$.ajax](http://api.jquery.com/jquery.ajax/)** comes from the [jQuery](http://jquery.com/) framework and it encapsulates the necessary code to handle AJAX calls with the server. There are two important things to remember about AJAX calls: * They are asynchronous: this is why callbacks are used to be notified when the answer comes back from the server * Like any function call, it may fail Consequently, there are two problems with the way the method **onClick** was implemented: * The use of **unfreezeInterface** is done right after the call to **$.ajax**. It means that it does not even wait for the call to succeed. * Furthermore, if the call fails, nothing will happen. The error is ignored.
