    If it's not tested, it does not work

    UI5 is shipped with testing tools :
    • qUnit: unit testing
    • OPA: component and integration testing

    One Page Acceptance Tests

    • UI behavior testing
    • JavaScript based
    • Tightly integrated with UI5
    • Asynchronous with active polling mechanisms
    • Backend is mocked

    Compared to other automation frameworks

    Puppeteer, Selenium or Webdriver.io are framework agnostic, one must rely on the generated HTML to code automation. They are designed for End 2 End testing.

    OPA is designed by and for UI5 developers: one manipulates UI5 controls.

    ðŸ†•WDI5 is built on top of Webdriver.io to enable UI5 controls selection.

    Executing qUnit & OPA tests

    • As simple as running a web page
    • A report documents execution results

    Continuous Integration & Delivery

    These tests can be automated in a pipeline to validate the code before merging and / or shipping the new version.

    Pipeline issues

    • CI pipeline uses the Karma runner
    • OPA tests based on the iFrame mode
    • Huge memory consumption
    • Tests are abruptly aborted (browser crash)

    Karma execution model

    • All tests in one window
    • One iFrame to run them all
    • Sequential execution
    • Coverage information

    The current situation

    Why is testing an issue ?

    • Development mode: use the browser
    • CI/CD pipeline: need a tool with heavy configuration

    The solution ?

    Put simply, ui5-test-runner is the fastest way to automate your tests :

    • Almost no configuration
    • Parallel execution of test pages

    Limits of version 1

    • Built as a POC
    • Problems are not easy to diagnose
    • Inbuilt server not as good as ui5/cli

    From v1 to v2

    Better feedback

    * interactive report * traces & network captured * configurable report generation (jsunit-xml)

    More browsers

    * puppeteer * selenium-webdriver (chrome, firefox, edge) * jsdom

    Better integration

    * legacy mode (explain why) * url mode (supports @ui5/cli, typescript or any special environment)
